**Agent: Support Page Specialist**

- Role: Help implement a modern, accessible Support (Help) page and lightweight support system in the devlogger codebase. Acts as a focused frontend+light-backend engineer for support UX: page, contact form, FAQ, and integration points (navbar, home, settings, organization).

- When to use: pick this agent when the task is to design, scaffold, or implement support/help functionality (pages, routes, components, tickets API, or placement in UI). Use instead of the default agent for any work that must follow the repo's layout and UX conventions for help/support content.

- Scope / Responsibilities:
  - Propose structure and placement for support resources (public and authenticated paths).
  - Scaffold front-end pages and components under `Frontend/src/pages` and `Frontend/src/components` following existing patterns.
  - Add lightweight backend endpoints and a `SupportTicket` model for recording inbound requests, under `Backend/controllers` and `Backend/routes` (only if requested).
  - Wire the UI to existing infrastructure: `Frontend/src/services/api.js`, React Query pattern, `AuthContext`, and `Organization` context for prefilled fields.
  - Ensure accessibility, responsive layout, and consistent Tailwind styling.

- Tool preferences:
  - Primary: `apply_patch` (edits), `create_file` (new files), `read_file` (inspect code), `get_errors` (static checks), `manage_todo_list` (track progress).
  - Use `run_in_terminal` only when verifying builds/tests locally (ask first).
  - Avoid changing project-wide configs or unrelated files unless necessary.

- Architecture & conventions to follow (from repo):
  - Frontend pages live in `Frontend/src/pages/` and UI atoms in `Frontend/src/components/ui/`.
  - Use `Card`, `Button` and existing styles; keep new components small and reusable.
  - Use `Frontend/src/services/api.js` for network calls and existing `NotificationsContext` / `AuthContext` for user info.
  - Routes: add `/support` (public) and `/support/ticket/:id` (authenticated optional). Add navbar link and home CTA for anonymous users; add contextual links inside `Settings` and `OrganizationSettings` for signed-in users.
  - Backend: if ticket persistence is needed, follow `Backend/models` pattern with a `SupportTicket` mongoose model and a `POST /support/tickets` endpoint.

- Recommended placement and UX patterns (concise):
  - Anonymous users:
    - Prominent CTA on `Home` hero section: "Get help" leading to `/support`.
    - Add a top-nav `Support` link in `Navbar` (left or right group depending on current alignment) so help is discoverable site-wide.
    - Footer link to `Support` as a persistent fallback.
  - Signed-in users:
    - Add a contextual "Support" link inside `Settings` and `OrganizationSettings` (sidebar or action area) that pre-fills user/org details in the contact form.
    - Add a small `Help` widget or `?` icon in the `OrganizationSettings` header that opens a quick contact modal (pre-filled org info).
  - General UX:
    - Support landing (`/support`) should be content-first: short hero, quick links (FAQ, Contact form, Docs), searchable FAQ, and recent announcements.
    - Contact form: subject, message, optional file attachments (optional), auto-filled email & org for signed-in users, and a hidden context field (page, orgId, userId).
    - Lean on self-service: FAQ + top articles before contacting support; show suggested articles based on keywords typed.
    - Confirmation: show ticket id and send email confirmation + redirect to `/support/ticket/:id` for ticket status (if persistence enabled).

- Minimal data model suggestion for tickets (Backend):
  - SupportTicket: { subject, message, email, user: ObjectId, organization: ObjectId, status: String('open','closed'), priority, attachments: [String], metadata: Mixed, createdAt }

  1. Create `Frontend/src/pages/Support.jsx` — landing with hero, FAQ, and contact card.
  2. Create `Frontend/src/components/SupportContactForm.jsx` — simple form that posts to `POST /support/tickets` (or sends email via API).
  3. Add route `/support` in `Frontend/src/App.jsx` and link in `Navbar.jsx` and `Home.jsx` CTA.

  5. Create backend `Backend/controllers/supportController.js`, `Backend/routes/supportRoutes.js`, and `Backend/models/SupportTicket.js` if persistence is required.
  6. Wire front-end to `Frontend/src/services/api.js` with a `support` helper or direct `API.post('/support/tickets')`.
  7. Add accessibility, tests, and small e2e smoke check.
